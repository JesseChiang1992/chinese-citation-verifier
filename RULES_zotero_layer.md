# RULES: Zotero 层处置规则（v2）

## 三态分流

| 状态 | 触发 | 标签 | 下一步 |
|---|---|---|---|
| **A · 完整命中** | Zotero search 命中 + 元数据完整判据全过 | `[已核验·Zotero·完整]` | 跳过 Step 2-3 网络层，直进 Step 5/6 |
| **B · 空壳待补全** | Zotero search 命中但元数据缺必要项 | `[已存在·Zotero·待补全]` | **暂停本条**，提示用户决策 |
| **C · 未命中** | search + advanced_search 均 0 条 | 不打标，沉默通过 | 进入 Step 2-3 网络层 |

判据见 `REFERENCE_Zotero.md`「元数据完整」段。

## A 态处置

- 网络层完全跳过——茉莉花已对接 CNKI / 万方 / 维普 / 中华医学杂志网，在 TCM / 国内队列域比 OpenAlex + Crossref 可靠
- 输出 schema：
  - `核验状态: 已核验·Zotero·完整`
  - `信源命中: [Zotero]`
  - `zotero_item_key: <8-char key>`（便于用户回查）
- 摘要从 `abstractNote` 取，**不**回查网络重组
- **特殊情况**：用户明确要求"再走网络层复核"时，可加跑 OpenAlex + Crossref，结果差异作为备注追加，不打 `冲突` 标签——Zotero/茉莉花是一手信源，OpenAlex 不一致时倾向信 Zotero

## B 态处置（最敏感）

- 标签：`[已存在·Zotero·待补全]` + `zotero_item_key`
- 列出 `缺失字段: [...]` 明细
- **铁律守护**：禁止用 title 凭空补 DOI / 作者 / 期刊 / 年——这等同于「凭记忆补全」，违反五条铁律之三
- 给用户两选一：
  1. **（默认推荐）** 在 Zotero 客户端右键该条目运行「茉莉花」插件富集元数据，富集完成后重试本核验
  2. 跳过 Zotero 层，让本 skill 走 v1 网络层（OpenAlex × Crossref，TCM 域命中率受限）
- 用户未明确选 → 默认走 (1)，本条**暂搁置**，**不**自动归入"未核验·信源失联"
- 在最终汇总中，B 态条目单列一节，标题为「待茉莉花富集后重核」

## C 态处置

- 不在报告中显式说"Zotero 未命中"——避免噪音
- 自然流入 Step 2-3 网络层
- 网络层全失败时按常规走 `[未核验·信源失联]`

## MCP 不可用（整层跳过）

触发：`zotero_list_libraries` 调用失败、超时，或工具不存在。

动作：
- 跳过 Step 0
- 报告顶部加一行：
  > 提示：未检测到 Zotero MCP（或工具不可用）。若文献已在本地 Zotero 中，配置 MCP 后命中率会显著提升（见 `SETUP_Zotero.md`）。
- 主流程继续走 v1 网络层，不阻断

## 与诚信门的一致性

- Zotero 信源等同于 CNKI / 万方（经茉莉花），属**用户本地侧的二手数据汇集**，技术上不构成"独立第二信源"——故 A 态使用 `已核验·Zotero·完整`，**不**升格为 `双源`
- 但其在 TCM / 国内队列域的可信度 **高于** OpenAlex+Crossref 双源（用户亲自收录 + 中文库直采），故跳过网络层是合理的优化，不是放松诚信门
- B 态绝不擅自补全——这是诚信门的核心边界，与 Zotero 层是否在场无关
